## 一、项目目标

为植百汇（健康空间服务商，服务近千家企业）搭建一套 7×24 小时微信智能客服系统，实现：

1. **客户身份识别**：来访客户自动确认属于哪个项目
2. **意图分类处理**：区分投诉、信息查询、日常咨询等场景
3. **业务系统联动**：通过 LeHome API 实时读写业务数据
4. **记忆持久化**：客户绑定项目后下次不用重复确认
5. **防封安全运行**：拟人化操作，长期稳定不封号

### 为什么选三层而不是直连 MySQL？

| 对比 | 直连 MySQL | 通过 API |
|------|-----------|---------|
| 数据一致性 | ⚠️ 绕过业务校验 | ✅ 走正规业务流程 |
| 安全性 | ⚠️ Python 端持有数据库密码 | ✅ API 做权限控制 |
| 维护性 | ⚠️ 表结构改了两边都要改 | ✅ API 做好版本控制即可 |
| 复用性 | ❌ 只有客服能用 | ✅ 小程序/APP 也能复用 |
| 开发量 | 少一些 | 多一点（但值得） |

#### 模块 A：LeHome API 客户端（lehome_client.py）

**职责**：封装对 LeHome API 的 HTTP 调用

```python
class LeHomeClient:
    """LeHome 业务 API 调用客户端"""
    
    def __init__(self, base_url="http://xxx:8092"):
        self.base_url = base_url
    
    # ===== 项目相关 =====
    def search_project(self, keyword) -> list:
        """模糊搜索项目"""
        return requests.get(f"{self.base_url}/api/wx/project/search", 
                          params={"q": keyword}).json()
    
    def get_project(self, project_id) -> dict:
        """获取项目详情"""
        return requests.get(f"{self.base_url}/api/wx/project/{project_id}").json()
    
    # ===== 工单相关 =====
    def get_repairs(self, project_id, limit=5) -> list:
        """查询最近工单"""
        return requests.get(f"{self.base_url}/api/wx/repair/list",
                          params={"projectId": project_id, "limit": limit}).json()
    
    def create_repair(self, project_id, type, desc, pic_url=None) -> dict:
        """创建投诉工单"""
        return requests.post(f"{self.base_url}/api/wx/repair/create",
                           json={"projectId": project_id, "type": type, 
                                 "desc": desc, "picUrl": pic_url}).json()
    
    # ===== 巡场相关 =====
    def get_patrols(self, project_id, limit=5) -> list:
        """查询巡场记录"""
        return requests.get(f"{self.base_url}/api/wx/patrol/list",
                          params={"projectId": project_id, "limit": limit}).json()
    
    # ===== 客户相关 =====
    def get_customer(self, customer_id) -> dict:
        """查询客户信息"""
        return requests.get(f"{self.base_url}/api/wx/customer/{customer_id}").json()
```

# ===== 工具执行器（AI 决定调哪个，这里负责执行）=====
class AgentToolExecutor:
    def __init__(self, lehome_client, binding_service):
        self.lehome = lehome_client
        self.binding = binding_service
    
    def execute(self, tool_name, arguments, chat_name):
        """执行 GPT 选择的工具，返回结果供 GPT 生成最终回复"""
        
        if tool_name == "search_project":
            return self.lehome.search_project(arguments["keyword"])
        
        elif tool_name == "bind_project":
            self.binding.bind_customer(
                chat_name, arguments["project_id"], arguments["project_name"])
            return {"success": True, "message": "绑定成功"}
        
        elif tool_name == "create_repair":
            binding = self.binding.get_binding(chat_name)
            return self.lehome.create_repair(
                binding["project_id"], arguments["type"], arguments["desc"])
        
        elif tool_name == "query_repairs":
            binding = self.binding.get_binding(chat_name)
            return self.lehome.get_repairs(binding["project_id"])
        
        elif tool_name == "query_patrols":
            binding = self.binding.get_binding(chat_name)
            return self.lehome.get_patrols(binding["project_id"])
        
        elif tool_name == "transfer_human":
            return self._create_group(chat_name, arguments["reason"])

**与意图标签方案的本质区别**：

```
意图标签（旧方案，已废弃）：
  用户消息 → GPT 回复 + 吐标签 → 我们的代码解析标签 → 调 API
  ↑ 我们决定有哪些意图，AI 只负责分类

Function Calling（新方案，Agent）：
  用户消息 → GPT 自主决定调哪个工具 → 执行工具 → GPT 拿到结果 → 生成回复
  ↑ AI 自主决策：要不要调工具、调哪个、传什么参数、怎么组织回复
```

## 六、Agent 消息处理循环

```
客户发来消息
    │
    ▼
┌─ SiverWXBot 原有流程 ──────────────────────────────────────┐
│  1. 消息去重                                                │
│  2. 是管理员命令？→ 执行命令                                 │
│  3. 关键词匹配？→ 直接回复                                   │
└──────────────────────────┬─────────────────────────────────┘
                           │ 未命中，进入 Agent 流程
                           ▼
┌─ 新增：上下文准备 ─────────────────────────────────────────┐
│                                                             │
│  4. 查绑定关系（本地 SQLite）                               │
│     → 已绑定：拿到 project_id + 项目信息                    │
│     → 未绑定：标记"新客户"                                  │
│                                                             │
│  5. 构建 System Prompt（注入客户状态和项目信息）             │
│  6. 加载对话记忆（最近 N 条）                               │
│                                                             │
└──────────────────────────┬─────────────────────────────────┘
                           │
                           ▼
┌─ 新增：Agent 循环（核心！）────────────────────────────────┐
│                                                             │
│  7. 调用 GPT-5.1（带 tools 参数 + tool_choice="auto"）      │
│                                                             │
│  8. GPT 返回结果，判断类型：                                │
│     │                                                       │
│     ├── 返回 tool_calls？（AI 决定调工具）                  │
│     │   → 执行工具（调 LeHome API / 写 SQLite）             │
│     │   → 将工具返回结果送回 GPT                            │
│     │   → 回到步骤 7（GPT 可能继续调工具或生成回复）        │
│     │   ↑ 这就是 Agent 循环！AI 可以连续调多个工具          │
│     │                                                       │
│     └── 返回 message？（AI 决定直接回复）                   │
│         → 得到最终回复文本                                   │
│                                                             │
└──────────────────────────┬─────────────────────────────────┘
                           │
                           ▼
┌─ SiverWXBot 原有流程 ──────────────────────────────────────┐
│  9.  human_delay() 随机延迟 3~8 秒                          │
│  10. 拆分多条发送（可选）                                    │
│  11. 发送回复                                               │
│  12. 存入对话记忆                                           │
└─────────────────────────────────────────────────────────────┘
```

### Agent 循环的威力（举例）

```
客户："我是金茂府的，前台绿植叶子黄了"

GPT 第1轮思考：
  → 客户提到了项目名，先搜索
  → tool_call: search_project(keyword="金茂府")
  → 得到结果：[{id: 123, name: "南京金茂府", level: 2}]

GPT 第2轮思考：
  → 找到项目了，客户语气很确定，帮他绑定
  → tool_call: bind_project(project_id=123, project_name="南京金茂府")
  → 得到结果：{success: true}

GPT 第3轮思考：
  → 绑定完了，客户还说了叶子黄的问题，帮他建工单
  → tool_call: create_repair(type="叶黄", desc="前台绿植叶子发黄")
  → 得到结果：{id: "R20260414001"}

GPT 第4轮：生成最终回复
  → "您好~已确认您是「南京金茂府」的客户[微笑]
     前台绿植叶黄的问题已记录，工单号 #R20260414001，
     养护师将在24小时内上门处理~"

一条消息，AI 自主连续调了 3 个工具！
这就是 Agent 和 Chatbot 的本质区别。
```

## 八、System Prompt 设计

```markdown
# 角色
你是植百汇的智能客服助手"小植"，一个温暖、专业的绿植养护顾问。
你有一系列工具可以使用，请根据对话情况自主判断是否需要调用。

# 公司介绍
植百汇是一家健康空间服务商，为企业提供绿植租赁、养护、室内景观设计等服务，
目前服务近千家企业客户。

# 客户状态
{customer_context}

# 你的工作原则
1. 客户未绑定项目 → 友好地询问项目名称，然后用 search_project 搜索
2. 搜索到项目后请客户确认 → 确认后用 bind_project 绑定
3. 客户反映植物问题 → 先共情安抚，然后用 create_repair 创建工单
4. 客户想查养护情况 → 用 query_repairs 或 query_patrols 查询
5. 问题超出你的能力 → 用 transfer_human 转人工
6. 日常聊天 → 直接友好回复，不需要调用工具

# 重要提醒
- 创建工单前，确保客户已绑定项目
- 如果客户未绑定就报问题，先引导绑定再建工单
- 工具调用的结果会自动返回给你，你根据结果生成回复
- 你可以在一次对话中连续调用多个工具

# 语气风格
- 温暖亲切，适当用"呢"、"哦"、"~"
- 偶尔用微信表情 [微笑][OK][抱歉]
- 单条不超过100字
- 投诉先共情，再解决
```

**注意：v3.0 的 Prompt 里不再需要定义意图标签规则，因为工具定义已经通过 `tools` 参数传给了 GPT，AI 自己知道什么时候该用什么工具。Prompt 更简洁、更自然。**
