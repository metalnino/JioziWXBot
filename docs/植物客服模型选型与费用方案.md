# 植物客服模型选型与费用方案

## 一、目标与范围

- 业务：植物客服机器人，覆盖图文问答、病虫害识别、养护建议
- 要求：按量计费优先，兼顾效果、响应速度和稳定性
- 平台：对接 12AI 与紫喵中转站的聊天/视觉模型

## 二、中转站信息

- 12AI 文档：[https://doc.12ai.org/](https://doc.12ai.org/)
- 紫喵文档：[https://apiconfig.aisonnet.org/public_html/zimiao.html](https://apiconfig.aisonnet.org/public_html/zimiao.html)

补充说明：

- 两站都可按 OpenAI 兼容方式接入
- 同一模型可能有“按 Token”与“按次 `-c`”两种形态
- 本方案统一使用“按 Token”模型（不带 `-c`）

## 三、推荐模型分层

### 1) 默认层（低成本常驻）

- 模型：`gemini-3-flash`
- 负责：常见养护问答、首轮图片初筛、引导补图/补充症状
- 原因：单位成本低，速度快，适合大多数客服请求

### 2) 增强层（复杂文本推理）

- 模型：`gpt-5-mini`
- 负责：复杂病因分析、多约束建议、长文本结构化输出
- 原因：效果明显高于轻量模型，成本仍可控

### 3) 兜底层（高精度视觉复核）

- 模型：`gemini-3-pro`
- 负责：多图细节复核、疑难样本二次判断、低置信度纠偏
- 原因：视觉理解更稳，适合小比例高价值请求

## 四、路由规则（可直接落地）

1. 用户含图片请求：先走 `gemini-3-flash`
2. 识别置信度低于阈值（建议 `< 0.72`）：升级 `gemini-3-pro`
3. 涉及农药剂量、可食用安全、宠物毒性：增加 `gpt-5-mini` 文本复核
4. 模型超时/失败：同模型重试 1 次后切换到另一站点同档模型
5. 连续两次低置信度或结论冲突：转人工

## 五、费用口径与计算公式

统一单位：每 `1M tokens`

基础公式：

`总费用 = 输入Token/1,000,000 * 输入单价 + 输出Token/1,000,000 * 输出单价`

若有缓存输入价：

`总费用 += 缓存输入Token/1,000,000 * 缓存输入单价`

说明：客服场景通常输出成本占比更高，比较中转站时优先看“输出单价”。

## 六、预算估算模板

设：

- 日请求量：`N`
- 平均输入：`Ti`
- 平均输出：`To`
- 升级率：`r`

则：

- 默认层日成本  
`C_base = N * (Ti * Pi_flash + To * Po_flash) / 1,000,000`

- 升级层日增量  
`C_upgrade = N * r * (Ti * Pi_upgrade + To * Po_upgrade) / 1,000,000`

- 总日成本  
`C_day = C_base + C_upgrade`

- 总月成本  
`C_month = C_day * 30`

## 七、中转站选择结论（当前版本）

在未补全两站同档模型完整单价前，先按以下策略执行：

- 成本优先：主用紫喵，默认层用 `gemini-3-flash`
- 质量优先：主用 12AI，增强层用 `gpt-5-mini`
- 稳定性优先：双站并行接入，按失败率自动切换

最终“只留一个站点”的判定标准：

1. 同档模型输出单价更低
2. 72 小时压测中超时率/5xx 更低
3. 视觉任务误判率更低

## 八、建议的初始生产配置

- 主站点：紫喵
- 备用站点：12AI
- 默认模型：`gemini-3-flash`
- 增强模型：`gpt-5-mini`
- 兜底模型：`gemini-3-pro`
- 转人工阈值：连续 2 次低置信度或 1 次高风险冲突

## 九、安全要求

- 不在仓库、日志、截图保留明文 API Key
- Key 仅放环境变量或密钥管理服务
- 对外展示时仅保留脱敏形式（如 `sk-xxxx...yyyy`）

---

如需最终“定版费用表”，把两站这 3 个模型的输入/输出单价填入本方案即可自动算出日/月成本并给出唯一站点选择。
